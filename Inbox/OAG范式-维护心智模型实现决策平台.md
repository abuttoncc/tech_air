# OAG 范式：维护心智模型 → 实现决策平台

> 这是一篇**母本框架笔记**。把 Palantir OAG 这套东西从产品话术里抽出来，提炼成我自己后续做"决策平台"时反复套用的骨架。不是案例复述，是可复用的设计范式。

## 一句话

> **交付目标 = 维护一个跟领域真实运转同构的心智模型（名词：世界是什么样），并在它之上实现一个能算、能推、能建议、能落地的决策平台（动词：那我该怎么办）。**

脱离业务事实的 AI 毫无价值。价值全在"双重锚定"：把 LLM 的智能架在**运营真相**（本体数据）和**数学真相**（ML 模型 / 逻辑工具）两根柱子上，幻觉的暴露面才收得住。

---

## 题眼：决策中心架构，不是数据中心架构

这篇（Palantir 首席架构师那篇）把整套东西的脊梁讲透了：**本体建模的对象是"决策"，不是"数据"。** 每个组织的头等使命是在内外条件不停变的情况下，尽可能做出最好的决策。传统数据架构只管报表和分析，**不捕获决策背后的推理，也不捕获决策落地后的动作**，所以它学不会、也接不进 AI——它跟运营是脱节的。要在今天赢，企业需要的是**决策中心的软件架构**。

一个决策拆成三要素，本体把这三样合进一个会随组织实时演化的统一模型：

| 要素 | 是什么 | 对应层 |
|---|---|---|
| **数据 Data** | 做决策用到的信息 | 名词层（语义 / semantic）|
| **逻辑 Logic** | 评估决策的过程 | —（连接名词与动词）|
| **行动 Action** | 决策的执行 | 动词层（动能 / kinetic）|

**名词 vs 动词**（这是干净的术语，值得抄）：本体里的数据元素是企业的"**名词**"——语义层的真实对象和关系；行动是"**动词**"——动能层的真实执行。每条本体驱动的工作流，都是把名词和动词凑成一个**完整句子**（中间嵌入各种逻辑）。光把数据统进语义模型有价值，光把逻辑拼齐能评估决策也有价值，但**只要执行出来的决策没跟运营系统同步，一切价值终归有限**。

AI 时代的主要矛盾不是"清洗统一数据"，而是**相关性 relevance**。最关键的一类数据是**决策数据**：一个决策当时的上下文、评估过的选项、最终选择的下游影响。端到端的"**决策血缘 decision lineage**"——某个决策在哪个时点、基于哪版数据、经哪个应用做出——被自动捕获，人和 AI 都能安全取用。这才是大规模喂养 AI 学习的底料。

## 两层骨架

### 第一层：心智模型（要维护）

不是知识库，不是文档堆，是领域的**数字孪生**：有对象、有关系、有状态、有时态。

- **同构**：跟真实领域怎么运转一一对应，能用手指指向"就是这一个"。
- **单一真相源**：一个 slug = 一个对象，跨来源去重，全库唯一。
- **活的**：靠"退役不删除 + 双时态"维护——旧值封存加版本，永不 DELETE。所以它永远回答"此刻这个领域到底什么样"。
- **腐烂即崩**：心智模型一旦失真，上面所有决策跟着废。维护是第一要务，不是建完就完。

### 第二层：决策平台（要实现）

在心智模型之上，让智能体把"是什么"变成"该怎么办"。执行回路：

```
用户意图
  → LLM 拆解意图、规划路径（指挥家，负责推理）
  → 碰到数学/预测/优化 → 调确定性的「逻辑工具」（不让 LLM 自己算）
  → 思维链全程留痕（实体提取→本体查询→工具调用→综合决策）
  → 输出建议 + 一个 Approve 按钮（人拍板）
  → 写回系统（Action，闭环）
```

核心重构（题眼）：**企业级 AI 不是让大模型做数学题，而是把它当"交响乐指挥家"——LLM 负责拆意图和推理，一碰到精确计算就精准调用确定性工具。** LLM 随机（stochastic、会幻觉、不擅算），逻辑工具确定（deterministic、算得准、但没上下文推理）。两者拼起来才完整。

## 关键零件：逻辑工具 = 一套端到端 MLOps（可复用设计）

一个"逻辑工具"不是一坨代码，是四层解耦的工程件——这是整套架构最硬、最值得抄的部分：

| 层 | 作用 | 纪律 |
|---|---|---|
| **建模目标** Modeling Objective | 业务终极诉求（如"预测客户订单量"） | **永久存在**，不随算法变 |
| **资产** Assets | 具体算法实现（Prophet / XGBoost / …） | **可热插拔替换**，目标层不动 |
| **模型适配器** Model Adapter | 抹平异构模型差异，吐标准输出 | **通用翻译器**，解耦的本质在这层 |
| **部署** Deployments | 真正跑起来的运行时 | 像 CI/CD：Sandbox→兼容检查→Staging→🔒人审→Production |

**适配器是题眼**：上游永远只看到统一格式（如 `日期 | 预测量 | 置信区间`），根本不关心底下换了什么模型。Palantir 给这套"一致接口接异构逻辑"起的名字叫**逻辑绑定 logic binding**——逻辑资产可能散在本地机房 / 企业云 / SaaS / 平台里，绑定层把它们统一成工作流可调的同一种接口。
**封装成工具只要几个算子**：`Input → Group By(本体聚合) → Map(转格式) → Transform(跑适配器) → Aggregate`。
**工具工厂飞轮**：开发者说意图 → AutoML 自动生成训练代码 → 训多种架构 → 胜出者变成新逻辑工具 →（用 AI 造 AI）。

## 关键零件：行动回路 = 安全写回 + 三权分立（补齐"动词"一柱）

> 这是原来那篇"下一篇预告 From Logic to Action"欠的账，这篇博客补上了。把决策闭合（写回运营系统）才是运营系统跟分析系统的分水岭。

数据是名词、逻辑是评估，**行动是把决策推回真实世界**。Action 这一柱的工程要点：

- **沙盒情景 Scenario**：模拟产生的结果先打包成"本体情景"——本体的一个**沙盒子集**，团队在里面安全推演这个决策的连锁影响（改这批原料会不会拖累同料生产的别的产品），**确认了再 commit**。等于给写回加了一道"先彩排"。
- **三权分立**（协作模型，直接抄）：能**探索**决策的人 / 能**暂存待审**的人 / 能**提交落地**的人，三种权限分开。谁能在什么条件下写回哪个系统，细到单字段。
- **按系统调写回**：同一个重分配方案落地时，仓储系统走 API 更新、三套 ERP 各走原生连接器（守各自的护栏）、生产计划系统收一个合并平文件异步消化。**写回例程一系统一调**，不是一把梭。
- **护栏 = 给 AI 划可行边界**：跟数据、逻辑同一套权限校验。默认 AI 只能把动作**暂存成情景**交人终审；但凭借细粒度日志和运营埋点，被验证过、跑熟了的 AI 流程可以**外科手术式地**获准自动闭环、免人审。**给 AI 的自由度可放可收，改了立刻在全库工作流生效。** copilot 像一个新团队成员，信任随表现逐步放权。

执行回路因此补全为：意图 → LLM 规划 → 调逻辑工具算 → 思维链留痕 → **结果先落沙盒情景推演** → 输出建议 + Approve → **按系统分别写回** → 监控系统响应、留审计。

## 决策中心学习闭环（危机之后才是真戏）

OAG 比普通 RAG 强的地方不止"接本体"，而是 LLM 能通过**可扩展的工具范式**同时接触**数据 / 逻辑 / 行动三类原语**，越过 RAG 只能接数据的天花板。而一旦数据、逻辑、行动全连进本体，组织就能做**决策中心的学习**：

- **决策血缘当训练燃料**：每个被评估过的数据元素、逻辑资产、动作都进端到端血缘；成千上万用户做出的决策聚合起来，可以安全地当微调训练数据。
- **部落知识蒸馏成 principles**：过去埋在工作流缝隙里、靠老员工口口相传的隐性知识，被 AI 照亮、提炼成一条条原则，在 LLM prompt 时被调用。**用 AI 改进 AI 的应用方式。**
- 解决一次危机顺带沉淀出的通用工作流，组织会想"记住"并在未来复用——这就是本体随时间增值的来路。

---

## 三条铁律（这才是交付难点，不是模型准不准）

1. **决策必须长在心智模型上，不能旁路。** LLM 一绕过本体自己拍脑袋，幻觉就回来了。每个预测、每条建议都要能溯回本体里的具体对象——这是 OAG 跟普通 RAG 拉开差距的唯一原因。
2. **可解释是信任的前提，不是锦上添花。** 每次实体识别、本体查询、工具调用都留痕（X-Ray 思维链）。业务敢把决策交出来，靠的是能看穿每一步，不是模型多准。
3. **人是最后的闸，不是被取代的对象。** 给的是"带算力的超级助手 + 一个 Approve 按钮"，控制权留在人手里。这条决定它能不能进关键业务。

## 镜像：我这套 `tech_air` 库本身就是这个范式的实例

不是在学外部案例，是给自己已经在搭的东西找到了母本：

| OAG | tech_air 对应物 |
|---|---|
| 心智模型 / 本体（名词层）| `wiki/` + 双时态 `data.db` + 退役不删除 |
| 逻辑工具 / 确定性后端 | auto-wiki 的 python 引擎（schema/store/expand）|
| 行动回路 / 安全写回（动词层）| `08-Ops/` 的写回：candidate → review → commit |
| 沙盒情景 Scenario | `review/` 审核队列（候选先进队列，不直写正典）|
| 三权分立（探索/暂存/提交）| 高危写入 gate：`newnode`/`xedge`/`t0-merge` 可升自动，`retire`/`disputed` 永久人工 |
| 自由度可放可收 | [[审批账本]]：连续批准升级一次驳回降级 |
| 决策血缘 decision lineage | `runs/` 运行记录 + ingest 的 `log.md` |
| 部落知识蒸馏成 principles | `pulse/` 今日启发束（心跳员产出）|
| "只投候选不直写正典" | 无人值守 agent 的产出收口原则 |

同一个骨架：Palantir 用本体+逻辑工具锚定企业真相，把决策推回 ERP；我用 auto-wiki 锚定领域认知，08-Ops 把候选推回 wiki 正典。**Action 那一柱的"写回 + 三权分立 + 自由度可放可收"，正好就是 08-Ops 的 gate 设计——这篇把我自己那套自动化层的合法性论证补全了。**

## 结论与行动

- 交付目标已锚定为两层：**维护心智模型**（本体，第一要务）+ **实现决策平台**（LLM 指挥家 + 逻辑工具 + 可解释 + 人审闭环）。
- 这篇是母本，且三支柱（Data/Logic/Action）已凑齐。后续任何"做一个决策平台"的设计，都从这张骨架起步：先问心智模型够不够同构、维护得活不活，再问决策回路的三条铁律守没守住，最后问行动那一柱的写回是否安全、三权是否分立。
- 下一步可选：① 用这张骨架审视 tech_air 现有 wiki/+08-Ops 的缺口（尤其 Action 一柱已能逐条对账）；② 针对一个真实场景从零设计落地架构；③ 框架已成熟，可正式 `/auto-wiki ingest` 把这套范式编译进 wiki/agent（建上面列的 OAG / 决策中心架构 / 写回模式 / 决策血缘 等节点）。
