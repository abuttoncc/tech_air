# Fable 5：指令与行为的呼应（基因型 ↔ 表现型）

> 关联：[[Fable5-系统提示词拆解]]（指令侧）· [[CL4R1T4S-生产级系统提示词合集]]（母资源）
> 外部参照（不在本库，只留指针）：`~/ablemind/able-alilab/.claude/skills/fable-ablework/SKILL.md`
> ——这份方法论，是从 Fable 5 在 ablework 代码库上的真实产出（ADR-001/002、framework-extraction、funnel/dispatch 访谈式 harness 维修）中逆向提炼而来，既是分析，也是决策指南。

## 核心关系

我们可以把同一个模型看作两面：

- **拆解笔记 → 基因型（genotype）**：这是**系统提示词**，代表我们下达给模型的**设计原则**。
- **fable-ablework SKILL → 表现型（phenotype）**：这是从模型**实际工作方式**里提炼出来的**行为方法论**，属于观察所得。

所谓“有没有呼应”，本质上是在问：**模型实际表现出的工作方式，能不能追溯回它被要求遵守的设计原则？**
先直接给结论：**性情层（disposition）高度呼应，工艺层（craft）则是模型在性情底座之上自己生长出来的增量。**

## 逐条对照

下表把 SKILL 中的核心方法论与系统提示词里的对应设计做了映射，右侧说明了这种呼应的本质。

| SKILL 方法论（从观察中提炼） | 拆解笔记对应（提示词里的设计） | 呼应的本质 |
|---|---|---|
| **§1 Layer Attribution**「先归层再动手，错层比不解更糟」 | 分层编排铁律 / L0–L6「上层不可被下层推翻」 | **同一种层级优先思维**——提示词按优先级分层编写，模型也本能地先把问题归到正确的层。几乎同构，这是最突出的一条。 |
| **§4 Evidence over Narrative**「怀疑 agent 自述，逐条对日志核验」 | L4 检索认识论（epistemics，如 164「不做 overconfident claims」）+「从不编造归因」(505) + wellbeing「理解依赖用户输入、无法验证」(96) + 抗幻觉技法 | **认知谦逊 + 反编造**。SKILL 指出“主脑会编造专业归因”，这正是提示词反复在防范的事。 |
| **§5 Conservative & Fail-Loud** | L1「感觉有风险就说更少」(36) + 安全 override 架构 + 红线「fail-open 抛异常」 | **保守默认 + 显式失败优于静默错误**，同一种安全姿态。 |
| **§7 Intellectual Honesty**「承认旧决策失效、诚实记 drift、抗过早收敛」 | L2 evenhandedness「结尾必附对立面，即便自己同意」(138) + mistakes「认错但不自我贬低」(152) | **把多个立场摆在同一张桌上，诚实优于表面一致性。** |
| **§8 Shrink Error Surface**「压平成标量、信息在最需要时可见」 | 技法「数值化硬阈值」「能力门控」+ 提示词自身的 `long_conversation_reminder`(130)、版权规则 4 处复述 | **降低歧义，在用到的地方呈现**——这是一种“元呼应”：模型如此设计自己的输出，正如它的提示词如此设计它。 |
| **§6 Scout First**「只读确认缺口边界再写码」 | L2「提到有文件不代表真有，自己 check」(80) + 产出文件前强制 `view` SKILL.md(307) | **先用只读手段验证现实，再动手修改。** |
| **产出物形态**：ADR 允许/不允许两栏 + 可复用决策树 | 技法「否定+正向替代成对」(384)、「条件路由决策树」(278)、「few-shot 正反对照」 | **模型偏好的文档形态，恰好是它自身提示词的编写方式。**又一个元呼应。 |
| **红线**：最小权限 / `WHERE user_id` / 跨租户 audit | L6「最小信任」沙箱（出网白名单 / 只读挂载）+ 能力门控 | **最小权限原则（least-privilege）**，同一种“时刻守卫”的本能。 |

## 两处“不呼应”——恰恰是关键

- **§2 框架抽取（依赖方向是第一性坏味道）** 与 **§3 外部服务决策树**：这些属于**软件架构的领域工艺**，系统提示词里**完全没有涉及**。它们不是提示词教的，而是模型将通用能力施加到代码领域时**自带的 craft**。
- **面向的场景不同**：SKILL 取自智能编码（agentic coding，即 Claude Code / Cowork）的场景，拆解笔记则面向 consumer chat（claude.ai）的系统提示词。`computer_use` / skills 层是二者之间的桥梁，但深层的架构方法论并不存在于 consumer 提示词中。

## 结论

**性情更像是内置的基因，工艺则是后天生长出来的能力。**

那 6 条性情级原则（层级思维、证据高于叙事、保守且响亮失败、诚实不轻率收敛、压缩出错面、只读先侦察）在“指令”和“观察”两侧同时出现，足以说明它们已经刻进模型的行为模式，并非某次任务的偶然表现。而框架抽取、服务分层这类，属于模型在性情底座之上、面对具体代码问题时自行补全的部分。

因此，`fable-ablework` SKILL 的价值并不在于复述系统提示词，而在于它捕获了系统提示词没有明说、模型却从性情中自然延伸出来的那一层——这正是“提炼方法论”时最有价值的地方。

## 启示（为自己搭建 agent / 编写 prompt）

- **想让 agent 稳定地表现出某种性情，就把它写进系统提示词的高优先级层**——表现型确实会回溯到基因型。
- **但不要指望提示词能直接教会领域工艺**——craft 来自模型能力与领域接触的乘积；我们能做的是通过 SKILL/skills 把提炼出的工艺固化下来复用（就像这份 fable-ablework 所做的那样）。
- **把“指令侧拆解”和“行为侧提炼”配成一对**，才能形成理解一个模型的完整闭环：一面看它被要求成为什么，一面看它实际成了什么。
