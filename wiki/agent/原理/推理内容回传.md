# 推理内容回传

推理型模型（带 thinking / reasoning）在多轮对话里，**历史推理内容如何跨轮回传**是一个独立于"答案内容"的生命周期问题。处理不当会同时引发两类隐蔽故障——这是 ablemind `REASONING_ROUNDTRIP_NOTES.md` 复盘出的硬教训：

**① fold 膨胀**：把 reasoning 并入出站 `content`（fold），会让每一轮历史都把推理正文重新塞进输入，多轮输入随之显著膨胀。修复（fold-fix）是让 reasoning 与 content **结构上同级独立**，绝不并入出站 content（在 `RouterChatModel` 层落实）。

**② 空签名回放死循环**：某些网关的 `thinking_signature` 为空——若把历史 thinking 当普通文本回传，模型会把它**当成新输入重读**，触发工具复读的死循环（曾有真实事故）。修复是 fail-loud：空签名网关上要开 thinking，必须显式声明 `thinking_roundtrip` 参数，缺省直接报错，而不是静默走错。

它教的耐用道理（durability: high）：**推理内容和答案内容是两条不同的生命周期，必须结构隔离、且其跨轮回传方式要按供应商签名显式处理**——否则就是上下文膨胀 + 工具死循环。这是"假设 → A/B 实测 → 推翻假设"的工程方法论范例，具体数值见来源的 NOTES 原文。
