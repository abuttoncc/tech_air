# 声明式智能体定义

把一个 agent 描述成**「它是什么」的声明（数据）**，而不是**「它怎么跑」的控制流（代码）**。执行循环——读消息、调模型、执行工具、把结果塞回上下文、再循环——由通用引擎统一负责，对所有声明一视同仁，不必每个 agent 重写。

## 题眼：声明 vs 编码

过去用 [[LangGraph]] 搭 agent，要自己写有向图状态机：定义 state、画节点、连边、写循环与分支——这是**命令式**，本质是一段程序，描述的是「执行过程」。

声明式把单位翻过来：一个 agent（或 subagent）就是几个字段——`name`（叫什么）/ `description`（干什么、何时被派活）/ `system_prompt`（人设与指令）/ `tools`（能用什么）/ `model`（哪个模型）。这是一张**规格说明书**，纯数据，没有一行讲先干嘛后干嘛。交给引擎，agent 就跑起来了。

[[deepagents]] 是这条范式的框架级落地：`create_deep_agent(subagents=[...])` 吃一组字典；Deep Agents Code 更进一步，让 subagent 就是磁盘上一个 `AGENTS.md` 文件——frontmatter 写 name/description/model，**正文整段即 system_prompt**，引擎启动时自动发现、自动挂上 `task` 工具。改 agent 行为 = 改一段文字，存盘即生效。

## 为什么是一个「模式」而非语法糖

关键是**关注点分离**：「能力声明」（agent 是什么）与「执行引擎」（怎么跑循环）解耦。引擎只写一次、人人共用；一百个不同 agent 的差异全在一百份声明里，引擎一份不变。由此带来的不是省几行代码，而是性质变化——

- agent 成为**可读、可 diff、可版本控制**的产物，review 它不需要懂图怎么编译，非工程师也能调它的脾气；
- 这是 SQL / HTML / Kubernetes manifest / Terraform 同一条老路在 agent 上的复用：**声明意图/状态，把达成过程交给运行时**。

它与 [[中间件分层编排]] 是同一套声明式哲学的两个面：后者讲**能力**以可插拔中间件按 yaml 声明装配，本模式讲**整个 agent 的身份**以声明描述。

## 边界：声明式优先，代码兜底

不是绝对无代码。当声明不够用——需要自定义控制流、复杂多步工作流——deepagents 留了 `CompiledSubAgent` 后门，可塞一个编译好的 [[LangGraph]] 图当 subagent。所以准确表述是「**声明式优先、代码兜底**」：默认你不该手写执行循环，少数确需自定义执行逻辑的才掉回去写图。
